# 01. 从暴力到优化

## 1. 算法不是从名字开始的

做题时不要一上来问：

```text
这题是什么算法？
```

更好的问题是：

```text
如果完全不优化，我会怎么做？
```

也就是先写出暴力思路。

暴力思路的价值不是为了提交，而是为了看清楚：

```text
我到底在枚举什么？
我在哪里重复做事？
我能不能把重复的东西保存下来？
```

## 2. 暴力法通常慢在哪里？

暴力法慢，通常不是因为代码丑，而是因为重复。

常见重复有几种：

```text
重复查找一个值
重复计算一段区间和
重复计算同一个子问题
重复往左/往右扫描
重复枚举相同意义的方案
```

优化就是找到这些浪费，然后换一种方式避免它。

## 3. 不同浪费对应不同工具

### 重复查找

如果暴力里一直在问：

```text
某个值以前出现过吗？
有没有 target - x？
```

通常可以想：

```text
HashMap
```

例子：

```text
Two Sum
```

### 重复计算区间和

如果暴力里一直在算：

```text
nums[i] + nums[i + 1] + ... + nums[j]
```

通常可以想：

```text
Prefix Sum
```

例子：

```text
Range Sum Query
Subarray Sum Equals K
```

### 重复计算子问题

如果递归里反复算同一个问题：

```text
f(n - 1)
f(n - 2)
```

通常可以想：

```text
DP / Memo
```

例子：

```text
Climbing Stairs
House Robber
```

### 重复往未来找答案

如果暴力里是：

```text
对每个 i，都往右边找某个最大值、最小值、下一个更大值
```

要问：

```text
能不能从右往左扫？
能不能维护一个 rightMax / rightMin？
能不能用 stack 存还没解决的元素？
```

例子：

```text
Best Time to Buy and Sell Stock
Daily Temperatures
Product of Array Except Self
```

### 重复枚举所有可能

如果题目本来就是要：

```text
所有方案
所有组合
所有排列
```

通常不是消灭枚举，而是有系统地枚举：

```text
Backtracking
```

重点是：

```text
做选择
递归
撤销选择
避免重复
```

## 4. 每题都可以问这几个问题

```text
1. 题目到底要我返回什么？
   一个值？数量？所有方案？是否存在？

2. 暴力法枚举什么？
   枚举一个点？两个点？一段区间？一条路径？一个选择序列？

3. 暴力为什么慢？
   重复查找？重复求和？重复子问题？重复扫描？

4. 我能保存什么？
   见过的值？
   累计结果？
   子问题答案？
   当前窗口状态？
   未来/过去的最值？

5. 信息在左边还是右边？
   如果需要左边信息，从左往右可能好维护。
   如果需要右边信息，从右往左可能好维护。
   如果左右都需要，可能要两遍。
```

## 5. 最重要的思维

算法优化经常来自这一句：

```text
我现在重复做了什么？
```

然后继续问：

```text
这个重复的东西能不能提前算？
能不能边走边维护？
能不能存起来下次直接用？
能不能换一个方向，让未来信息变成已经见过的信息？
```

先回答这些问题，再给算法命名。
