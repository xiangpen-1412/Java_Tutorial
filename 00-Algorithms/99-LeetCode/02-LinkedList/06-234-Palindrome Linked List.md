# LeetCode 234：Palindrome Linked List 复盘总结

## 题目

给一个单链表 `head`，判断这个链表从前往后读和从后往前读是不是一样。

如果一样，就是 palindrome linked list。

重点是：

```text
链表不能像数组一样直接从尾部往前访问。
```
## 这道题我一开始的问题

这道题表面上是判断链表是不是回文，但我真正卡住的不是“回文”这个概念，而是：

```text
1. 快慢指针结束后 slow / fast 分别代表什么
2. 奇数长度和偶数长度怎么处理
3. 反转链表时指针移动顺序为什么不能乱
4. 改 cur.next 之前为什么必须先保存 next
5. 为什么会出现 NullPointerException
```

这道题暴露出来的核心问题是：  
我对链表的理解还停留在“节点一个一个往后走”，但还没有完全形成“我是在改箭头”的意识。

---

## 错误一：while 条件用了 `||`

我一开始写过类似这样的判断：

```java
while (fast != null || fast.next != null)
```

这个是错的。

问题在于：

```text
当 fast == null 的时候，Java 还会去判断 fast.next != null
```

于是就会直接空指针。

正确的思维应该是：

```text
fast 每次要走两步，所以必须保证：
1. fast 自己不是 null
2. fast.next 也不是 null
```

所以这里必须用：

```java
while (fast != null && fast.next != null)
```

这次我学到的是：

```text
只要我要访问 xxx.next，就必须先保证 xxx != null。
```

链表题里，空指针不是偶然错误，而是边界条件没有想清楚。

---

## 错误二：fast 走完之后，我又马上访问了 `fast.next`

我后来虽然把 while 条件改成了：

```java
while (fast != null && fast.next != null)
```

但我在循环里面又写了类似：

```java
fast = fast.next.next;

if (fast.next.next == null) {
    ...
}
```

这个还是会炸。

原因是：

```text
fast = fast.next.next 执行完之后，fast 本身可能已经变成 null 了。
```

所以后面马上访问：

```java
fast.next
```

就等于访问：

```java
null.next
```

这次我学到的是：

```text
不是 while 入口安全，while 里面就永远安全。
```

尤其是当我改变了某个指针的位置之后，必须重新判断它是不是 null。

---

## 错误三：我试图在循环中手动判断奇偶长度

我一开始想用：

```java
boolean isOdd = true;
```

然后在快慢指针移动过程中手动判断链表是奇数长度还是偶数长度。

这个思路把问题搞复杂了。

后来我才理解：

```text
快慢指针结束后，fast 的状态天然就能告诉我链表长度是奇数还是偶数。
```

规则是：

```text
fast == null     偶数长度
fast != null     奇数长度
```

因为 fast 每次走两步：

```text
偶数长度：fast 正好走出链表，变成 null
奇数长度：fast 停在最后一个节点
```

这次我学到的是：

```text
有些信息不需要额外变量记录，算法运行结束后的指针状态本身就是答案。
```

---

## 错误四：没有真正理解 slow 最后停在哪里

我一开始对 `slow` 的位置不够清楚。

快慢指针结束后：

### 偶数长度

比如：

```text
1 -> 2 -> 2 -> 1
```

结束后：

```text
slow 指向后半段的开头
```

也就是第二个 `2`。

所以偶数长度可以直接：

```java
reverse(slow)
```

---

### 奇数长度

比如：

```text
1 -> 2 -> 3 -> 2 -> 1
```

结束后：

```text
slow 指向中间节点 3
```

中间节点不需要比较，所以要从：

```java
slow.next
```

开始反转。

这次我学到的是：

```text
快慢指针不是只会“找中点”，还要搞清楚中点到底是哪一个节点。
```

尤其是链表长度为奇数和偶数时，`slow` 的含义不同。

---

## 错误五：反转链表时写了错误的 `next.next = cur`

我在反转链表时写过类似这样的代码：

```java
ListNode next = cur.next;
cur.next = prev;
next.next = cur;
cur = next;
```

这里最大的错误是：

```java
next.next = cur;
```

这句不应该出现在标准反转链表里。

问题有几个：

```text
1. next 可能是 null，访问 next.next 会空指针
2. 反转链表只需要改 cur.next，不需要改 next.next
3. 这句可能把链表接乱，甚至制造环
4. 我没有正确更新 prev，导致已经反转好的部分没有被维护起来
```

这次我学到的是：

```text
反转链表每一轮只处理当前节点 cur。
```

当前这一轮只需要做一件核心事情：

```java
cur.next = prev;
```

也就是把当前节点的箭头反过来。

---

## 错误六：反转链表时忘记移动 `prev`

反转链表里，`prev` 的含义是：

```text
已经反转好的那一段链表的头节点
```

每处理完一个 `cur`，就必须让：

```java
prev = cur;
```

否则 `prev` 一直是 `null`，反转好的链表根本没有被维护起来。

这次我学到的是：

```text
prev 不是摆设，它代表“已经完成的结果”。
```

如果不更新 `prev`，最后就没有新的头节点可以返回。

---

## 错误七：对“保存后路”的重要性理解不够

反转链表中最重要的一句是：

```java
ListNode next = cur.next;
```

它必须在：

```java
cur.next = prev;
```

之前执行。

因为一旦我改了：

```java
cur.next
```

原来后面的链表就可能找不到了。

这次我学到的是：

```text
凡是我要改 cur.next，改之前必须先保存原来的 cur.next。
```

这就是链表反转里的“保存后路”。

---

## 错误八：比较两个半段时，我一开始没意识到只需要看后半段

链表后半段反转之后，只需要比较：

```text
head 开始的前半段
reversed 开始的后半段
```

循环条件只需要看：

```java
while (reversed != null)
```

因为 `reversed` 的长度就是需要比较的长度。

我一开始写成：

```java
while (reversed != null && head != null)
```

虽然有时候也能跑，但这个条件没有抓住重点。

这次我学到的是：

```text
比较多少个节点，由后半段 reversed 的长度决定。
```

## 这道题我真正学到的东西

### 1. 链表题不是操作值，而是操作箭头

数组题很多时候是在比较值、移动下标。

但链表题的核心是：

```text
节点之间靠 next 连起来
```

所以链表题最重要的是：

```text
我现在改了谁的 next？
改完之后，后面的链还能不能找到？
有没有可能形成环？
有没有可能访问 null.next？
```

---

### 2. 快慢指针结束后的状态很重要

快慢指针不是机械地写模板。

我需要知道：

```text
slow 最后在哪里
fast 最后是 null 还是非 null
这个状态分别代表什么
```

在这道题里：

```text
fast == null     偶数长度，slow 是后半段开头
fast != null     奇数长度，slow 是中间节点
```

---

### 3. 空指针错误本质上是边界没想清楚

这道题里报错多次，本质都是因为我访问了：

```java
xxx.next
```

但没有保证：

```java
xxx != null
```

以后只要看到：

```java
node.next
node.next.next
```

我就要立刻反问自己：

```text
node 一定不是 null 吗？
node.next 一定不是 null 吗？
```

---

### 4. 反转链表必须形成肌肉记忆

反转链表四步必须非常熟：

```java
ListNode next = cur.next;
cur.next = prev;
prev = cur;
cur = next;
```

对应理解是：

```text
先存后路
再改箭头
prev 前进
cur 前进
```

这四步顺序不能乱。

---

### 5. 不要额外制造没必要的变量

我一开始想用 `isOdd` 手动记录奇偶。

但后来发现：

```text
fast 本身的状态已经能表达奇偶。
```

所以算法题里不要急着加变量。

先想：

```text
当前已有指针的状态能不能直接提供答案？
```

---

## 我这道题的最大收获

这道题让我真正意识到：

```text
链表题最难的不是代码长，而是每个指针此刻到底指向哪里。
```

以后写链表题，不能只凭感觉写。

每次改 `next` 之前，都要在脑子里过一遍：

```text
1. 我有没有保存后面的节点？
2. 我现在要改的是谁的 next？
3. 改完之后，当前节点应该接到哪里？
4. prev 和 cur 应该怎么移动？
5. 有没有可能访问 null.next？
```

这道题的价值不是“会判断回文链表”，而是把这几个链表基本功串起来了：

```text
快慢指针
找中点
奇偶长度处理
反转链表
空指针边界
链表比较
```

真正要记住的是：

```text
链表题里，每一个 next 都必须有理由。
```
