# String 已做题目总结：固定长度滑动窗口与字符频率

当前 `04-String` 只有 [[01-567-Permutation in String]]。因此这份总结只覆盖已经学过的固定长度窗口、频率数组和增量维护，不扩展到尚未记录的字符串算法。

## 1. 当前覆盖的题型

| 题目 | 题目类型 | 主要方法 | 核心状态 |
| --- | --- | --- | --- |
| [[01-567-Permutation in String]] | 判断某个固定长度子串是否为排列 | 固定长度滑动窗口 + 频率表 | 模式串频率与窗口频率 |

## 2. 题型本质

排列不要求字符顺序相同，只要求字符种类和出现次数相同。

假设 `s1.length() == m`，需要判断 `s2` 中是否存在一个长度恰好为 `m` 的连续窗口，使：

```text
窗口字符频率 == s1 字符频率
```

所以这里同时出现两个信号：

```text
连续子串 → 滑动窗口
排列判断 → 字符频率
```

## 3. 三个版本代表的优化过程

### 每个窗口排序

将每个长度为 `m` 的子串排序，再和排序后的 `s1` 比较。逻辑直观，但每个窗口都重复截取和排序。

### 每个窗口重新统计频率

排序被长度固定的频率数组替代，但每移动一次窗口仍然从头统计全部字符。

### 增量维护频率

窗口向右移动一格时，绝大部分字符没有变化：

```text
移出左侧字符 → 对应频率减一
加入右侧字符 → 对应频率加一
```

这才真正利用了相邻窗口的重复信息。

## 4. 固定长度窗口的结构

```text
先初始化第一个长度为 m 的窗口
比较频率
之后每次移动：删除一个旧字符，加入一个新字符
再次比较频率
```

如果：

```text
s1.length() > s2.length()
```

则 `s2` 中不可能存在足够长的窗口。

## 5. 为什么频率数组适合这里？

当前题目字符范围固定且较小，可以使用：

```java
int[] frequency = new int[26];
```

字符到下标的映射：

```java
int index = character - 'a';
```

相比 HashMap，数组访问更直接；相比排序，频率表更贴近“排列只关心数量”的本质。

## 6. 如何识别同类题

```text
要求在一个长字符串中寻找某种短字符串模式；
答案必须是连续子串；
窗口长度由模式串长度固定；
顺序不重要，但字符数量重要。
```

看到这些条件，优先考虑：

```text
固定长度滑动窗口 + 字符频率表
```

## 7. 常见错误

```text
把排列理解成完全相同的字符串顺序；
substring 的右边界忘记是不包含；
窗口右移时只加入新字符，没有删除旧字符；
第一个窗口长度不足；
更新窗口后在错误的时机比较；
虽然写了 left/right，却仍然每轮重新统计整个窗口。
```

## 8. 记忆主线

```text
排列比较的是频率，不是顺序。
连续子串使用窗口。
窗口长度由 s1 决定。
相邻窗口只变化两个字符，所以增量更新。
```
