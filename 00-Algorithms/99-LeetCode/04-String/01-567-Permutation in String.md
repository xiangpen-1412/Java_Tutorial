# 567. Permutation in String：解法优化过程

## 题目本质

题目要求判断：

> `s2` 中是否存在一个长度等于 `s1.length()` 的连续子字符串，并且该子字符串与 `s1` 包含完全相同的字符和字符数量。

这道题有两个核心特征：

```text
1. 检查的是连续子字符串
2. 检查窗口长度固定为 s1.length()
```

因此可以使用：

```text
固定长度滑动窗口
```

但使用滑动窗口并不代表一定高效，关键还要看：

> 每次窗口移动后，我们如何判断当前窗口是否匹配。

---

# 第一版：截取窗口并排序

最初的思路是：

```text
1. 在 s2 中取出长度等于 s1.length() 的窗口
2. 把窗口中的字符排序
3. 把 s1 排序
4. 比较两个排序后的字符串
```

核心过程：

```java
String cur = s2.substring(left, right + 1);

char[] curArray = cur.toCharArray();
Arrays.sort(curArray);

String sortedWindow = new String(curArray);
```

这种方法的逻辑是正确的。

因为两个字符串如果互为排列，那么排序后的结果一定相同：

```text
"ab" → "ab"
"ba" → "ab"
```

## 这一版学到的东西

### 1. 排列问题可以转换成标准形式比较

字符顺序不同，但是字符组成相同：

```text
"eat"
"tea"
"ate"
```

排序后都会变成：

```text
"aet"
```

这也是 `Group Anagrams` 常用排序作为 HashMap key 的原因。

### 2. `substring()` 的右边界不包含

```java
s2.substring(left, right + 1);
```

如果要包含 `right`，结束位置必须写成：

```java
right + 1
```

### 3. `String` 不能直接排序

需要完成以下转换：

```text
String
→ char[]
→ Arrays.sort()
→ String
```

代码：

```java
char[] chars = str.toCharArray();
Arrays.sort(chars);
String sorted = new String(chars);
```

---

# 第一版为什么慢

虽然窗口长度固定，但每个窗口都进行了完整排序：

```text
截取字符串
→ 创建 char[]
→ 排序
→ 创建新 String
→ 比较
```

假设：

```text
s1.length() = m
s2.length() = n
```

一共有大约：

```text
n - m + 1
```

个窗口。

每个窗口排序需要：

```text
O(m log m)
```

总时间复杂度接近：

```text
O(n × m log m)
```

问题不在于排序本身不能用，而在于：

> 大量高度重叠的窗口被反复重新排序。

例如：

```text
"eid"
 "idb"
```

两个窗口的大部分字符是重复的，但排序方法没有利用之前窗口的计算结果。

---

# 第二版：使用频率数组代替排序

为了避免排序，可以记录每个字符出现的次数。

因为题目明确说明：

```text
字符串只包含小写英文字母
```

字符范围只有：

```text
a ~ z
```

因此可以使用长度为 26 的数组：

```java
int[] need = new int[26];
int[] window = new int[26];
```

其中：

```text
need   → 记录 s1 中每个字符出现的次数
window → 记录当前窗口中每个字符出现的次数
```

字符与数组下标的对应关系：

```text
0  → a
1  → b
2  → c
...
25 → z
```

统计字符：

```java
need[s1.charAt(i) - 'a']++;
```

这行代码可以拆成：

```java
char currentChar = s1.charAt(i);
int index = currentChar - 'a';
need[index]++;
```

---

# 第二版的问题：每个窗口仍然重新统计

第二版虽然不再排序，但每个窗口仍然执行：

```java
String substring = s2.substring(left, right + 1);

for (int i = 0; i < substring.length(); i++) {
    window[substring.charAt(i) - 'a']++;
}
```

检查完后又重新创建数组：

```java
window = new int[26];
```

整个过程仍然是：

```text
窗口移动一次
→ 截取整个窗口
→ 遍历整个窗口
→ 重新统计全部字符
→ 清空统计结果
```

假设窗口长度为 `m`，每个窗口都要重新遍历 `m` 个字符。

总时间复杂度大约是：

```text
O(n × m)
```

相比排序版有所改进，但仍然没有真正利用滑动窗口的特点。

## 这一版学到的东西

> 换了数据结构，不代表真正优化了算法。

虽然从排序换成了频率数组，但是如果每轮仍然重新计算全部内容，就仍然存在大量重复工作。

真正需要优化的是：

```text
不要重新计算整个窗口
```

而不是单纯把：

```text
排序
```

换成：

```text
计数
```

---

# 第三版：增量维护频率表

固定长度滑动窗口每次向右移动一格时，只发生两个变化：

```text
1. 右边进入一个新字符
2. 左边离开一个旧字符
```

例如：

```text
当前窗口："ei"
下一个窗口："id"
```

变化只有：

```text
e 离开
d 进入
```

因此不需要重新统计整个 `"id"`。

只需要：

```java
window['e' - 'a']--;
window['d' - 'a']++;
```

对应通用代码：

```java
window[s2.charAt(right) - 'a']++;
window[s2.charAt(left) - 'a']--;
left++;
```

这叫：

```text
增量维护状态
```

也就是只根据本次发生的变化，更新之前已经保存的结果。

---

# 真正的滑动窗口思想

滑动窗口不只是让两个指针移动。

真正的核心是：

> 窗口移动时，复用上一个窗口的状态，而不是重新计算整个新窗口。

错误理解：

```text
滑动窗口 = left 和 right 向右移动
```

更完整的理解：

```text
滑动窗口 =
维护一个连续区间
+ 记录当前区间的状态
+ 加入新元素时更新状态
+ 移除旧元素时更新状态
```

在这道题中：

```text
窗口状态 = 每个字符出现的次数
```

更新状态：

```java
window[incoming - 'a']++;
window[outgoing - 'a']--;
```

---

# 第一个窗口需要单独初始化

可以先统计 `s1` 和 `s2` 的第一个窗口：

```java
for (int i = 0; i < s1.length(); i++) {
    need[s1.charAt(i) - 'a']++;
    window[s2.charAt(i) - 'a']++;
}
```

初始化后，第一个窗口已经存在，必须先进行一次比较：

```java
if (Arrays.equals(need, window)) {
    return true;
}
```

否则可能漏掉第一个窗口。

例如：

```text
s1 = "ab"
s2 = "abxxxx"
```

第一个窗口 `"ab"` 已经匹配。

---

# 为什么后续 right 从 s1.length() 开始

假设：

```text
s1.length() = 2
```

第一个窗口已经统计了：

```text
s2[0]
s2[1]
```

因此后续要进入窗口的第一个新字符是：

```text
s2[2]
```

也就是：

```java
right = s1.length();
```

不能从：

```java
s1.length() - 1
```

开始，否则最后一个字符会被重复加入。

错误过程：

```text
第一个窗口已经包含下标 1
循环又从下标 1 开始
→ 下标 1 被统计两次
```

---

# 边界条件

如果：

```text
s1.length() > s2.length()
```

那么 `s2` 中不可能存在长度等于 `s1.length()` 的窗口。

因此需要提前返回：

```java
if (s1.length() > s2.length()) {
    return false;
}
```

这不仅是业务判断，也可以防止初始化第一个窗口时数组越界。

# 三个版本的对比

## 第一版：每个窗口排序

```text
每个窗口：
substring
→ char[]
→ sort
→ String
→ compare
```

时间复杂度：

```text
O(n × m log m)
```

---

## 第二版：每个窗口重新统计频率

```text
每个窗口：
substring
→ 遍历整个窗口
→ 创建频率表
→ compare
```

时间复杂度：

```text
O(n × m)
```

---

## 第三版：维护窗口频率

```text
初始化第一个窗口一次

后续每次：
加入右边字符
移除左边字符
比较频率表
```

时间复杂度：

```text
O(n × 26)
```

因为 `26` 是固定常数，所以通常写成：

```text
O(n)
```

空间复杂度：

```text
O(26) = O(1)
```

# 为什么 Group Anagrams 可以排序，这道题却更适合频率表

`Group Anagrams` 中，每个字符串相互独立：

```text
"eat"
"tea"
"tan"
```

每个字符串通常只排序一次，因此排序作为 HashMap key 很自然。

但这道题中的窗口高度重叠：

```text
"eid"
"idb"
"dba"
"bao"
```

每次只变化两个字符。

所以这里存在可以复用的状态：

```text
上一个窗口的字符频率
```

区别不在于：

```text
排序什么时候快，什么时候慢
```

而在于：

```text
同一个昂贵操作需要重复执行多少次
以及之前的计算结果能否复用
```

# 这道题真正应该掌握的知识

## 1. 固定长度窗口

当题目要求检查：

```text
所有长度相同的连续子字符串或子数组
```

可以考虑固定长度滑动窗口。

---

## 2. 频率表

当题目关心：

```text
每个元素出现了多少次
```

而不关心元素原始顺序时，可以考虑频率表。

选择方式：

```text
范围小且固定 → 数组
范围大或未知 → HashMap
```

---

## 3. 增量更新

窗口移动时，不要重新计算整个窗口。

应该思考：

```text
哪个元素进入了？
哪个元素离开了？
状态应该如何增加或减少？
```

通用形式：

```java
state[incoming]++;
state[outgoing]--;
```

---

## 4. 区分“使用了滑动窗口”和“真正优化了滑动窗口”

仅仅拥有：

```java
left
right
```

并不代表算法已经高效。

还要检查：

```text
窗口移动后，是否仍然从头计算窗口内容？
```

如果答案是“是”，通常仍有优化空间。

---

# 通用识别模板

以后遇到新题，可以依次问：

```text
1. 题目检查的是不是连续区间？
2. 窗口长度是固定还是可变？
3. 窗口里需要维护什么状态？
4. 新元素进入时，状态如何更新？
5. 旧元素离开时，状态如何更新？
6. 能否复用上一个窗口的结果？
```

如果题目具有：

```text
固定长度
+ 连续区间
+ 比较元素数量
```

可以优先考虑：

```text
固定长度滑动窗口 + 频率表
```

---

# 最重要的总结

这次优化过程不是：

```text
记住 Permutation in String 要用 int[26]
```

而是理解：

```text
排序解决了“顺序不同”的问题
频率表解决了“字符次数”的问题
滑动窗口解决了“连续区间”的问题
增量更新解决了“重复计算”的问题
```

真正的思维升级是：

```text
从每次重新计算整个窗口
变成只处理窗口发生的变化
```
