# LeetCode 310：Minimum Height Trees

> 难度：Medium  
> 主题：无向树、树的中心、Leaf Trimming、按层 BFS、Degree

## 1. 题目在问什么？

给定一棵包含 `n` 个节点的无向树，可以选择任意节点作为根。

选择节点 `root` 后，树的高度是：

```text
root 到所有节点的距离中的最大值
```

也就是：

```text
height(root) = max(distance(root, 其他所有节点))
```

题目要求找出所有能够让这个最大距离最小的根。

换一种说法：

> 在树中寻找一个或两个最居中的节点，让它们到最远节点的距离尽可能短。

## 2. 为什么第一眼不容易想到？

这棵树一开始没有固定的根。题目允许将每个节点分别作为根，因此根发生变化时，整棵树的父子方向也会变化。

但树本身的无向连接关系始终不变。

例如：

```text
0 — 1 — 2
```

以 `0` 为根时可以画成：

```text
0
└── 1
    └── 2
```

以 `1` 为根时可以画成：

```text
  1
 / \
0   2
```

所以这题不能依赖图片中的上下位置，而要把输入理解成一张无向图。

## 3. 从暴力解法开始

最直接的思路是：

```text
将每个节点分别作为 root
→ 从 root 遍历整棵树
→ 计算 root 到最远节点的距离
→ 比较所有 root 的 height
→ 保留产生最小 height 的节点
```

这个思路完全正确。

对于一棵树，一次 BFS 或 DFS 可以计算某个根的高度，时间复杂度是 `O(n)`。如果对 `n` 个根分别执行一次：

```text
时间复杂度 = O(n²)
```

暴力思路的价值不只是得到一个慢答案，而是明确了优化目标：

> 能不能不为每个根都重新遍历整棵树，就提前排除不可能最优的节点？

## 4. 第一个核心观察：最远节点一定是叶子

假设从某个根出发，最远节点 `x` 不是叶子，那么还可以从 `x` 继续走向另一个节点：

```text
root → ... → x → y
```

这样 `y` 就会比 `x` 更远，与“`x` 已经是最远节点”矛盾。

因此：

> 对任意一个根，决定树高的最远节点一定出现在树的外围叶子上。

树高不是由根附近的局部连接决定的，而是由整棵树最远的边缘决定的。

这也说明不能简单地寻找 degree 最大的节点：degree 只描述局部连接数量，height 描述全局最远距离。

## 5. 第二个核心观察：外围叶子不可能优于它的邻居

假设 `leaf` 是叶子，唯一邻居是 `neighbor`：

```text
leaf — neighbor — ...
```

如果选择 `leaf` 作为根，前往其他任何节点都必须先经过 `neighbor`：

```text
leaf → neighbor → 其他节点
```

与其从 `leaf` 出发，不如直接从 `neighbor` 出发。对于其他方向上的节点，距离都会减少一步。

因此，当树中还有超过两个节点时：

> 当前最外围的叶子可以直接从最佳根候选中排除。

这就是从暴力解法走向优化的第一步：不再真的计算每个叶子的完整高度。

## 6. 第三个核心观察：删除一层叶子后，相对优劣不变

考虑：

```text
0 — 1 — 2 — 3 — 4
```

原树中部分节点的高度：

```text
root 1：height = 3
root 2：height = 2
root 3：height = 3
```

排除外围叶子 `0、4` 后：

```text
1 — 2 — 3
```

剩余节点在新树中的高度：

```text
root 1：height = 2
root 2：height = 1
root 3：height = 2
```

所有剩余节点的高度统一减少了 `1`，但它们之间的相对关系没有改变：

```text
原树中 2 最优
缩小后的树中 2 仍然最优
```

因此可以将问题不断缩小：

```text
在原树中寻找中心
→ 排除当前所有外围叶子
→ 在更小的树中寻找中心
→ 再排除新产生的外围叶子
→ 最后留下树的中心
```

## 7. Leaf Trimming 是什么？

Leaf Trimming 可以翻译为叶子剥离或叶子修剪。

它的过程是：

```text
找到当前所有叶子
→ 同时排除这一整层叶子
→ 更新它们邻居的 degree
→ 找到新产生的叶子
→ 继续从外向内收缩
```

它像剥洋葱一样，一层一层去掉外围，直到只剩下最中心的区域。

例如：

```text
0 — 1 — 2 — 3 — 4
    1 — 2 — 3
        2
```

偶数个节点的链会留下两个中心：

```text
0 — 1 — 2 — 3 — 4 — 5
    1 — 2 — 3 — 4
        2 — 3
```

所以一棵树的中心可能是：

- 一个节点；
- 两个相邻节点。

## 8. “外围叶子”与图片形状无关

无向树中的叶子通过 degree 判断：

```text
degree == 1
```

它不是指“图片中画在最下面的节点”。同一棵无向树可以选择不同节点放在图片顶部，但节点之间的连接关系不会改变。

例如：

```text
3 — 0
3 — 1
3 — 2
3 — 4
4 — 5
```

degree 是：

```text
degree[0] = 1
degree[1] = 1
degree[2] = 1
degree[3] = 4
degree[4] = 2
degree[5] = 1
```

因此第一层叶子是：

```text
0、1、2、5
```

删除它们后：

```text
degree[3]: 4 → 1
degree[4]: 2 → 1
```

剩余结构是：

```text
3 — 4
```

这里的叶子始终根据当前剩余结构中的 degree 判断，而不是根据画法判断。

## 9. 为什么要建立无向邻接表？

输入 `edges` 是边列表：

```java
edges[i] = {a, b}
```

表示无向边：

```text
a — b
```

Leaf Trimming 在删除一个叶子时，需要快速知道它连接了哪个邻居。因此要将边列表转换为邻接表。

因为是无向边，必须双向保存：

```java
rel.get(a).add(b);
rel.get(b).add(a);
```

同时统计两端的 degree：

```java
degrees[a]++;
degrees[b]++;
```

最终：

```text
rel.get(node)  → node 的所有邻居
degrees[node]  → node 当前还剩多少条连接
```

## 10. 不需要真的从邻接表删除边

删除一个叶子时，不需要真的修改 `rel` 中的数组元素。

只需要对它的邻居执行：

```java
degrees[target]--;
```

这表示连接当前叶子的那条边已经被逻辑删除。

如果邻居的 degree 变成 `1`：

```java
if (--degrees[target] == 1) {
    queue.offer(target);
}
```

说明这个邻居已经成为下一层的新叶子。

这种思想非常通用：

> 当实际删除数据结构中的边比较麻烦时，可以通过维护剩余 degree 表示逻辑删除。

## 11. 为什么必须按层 BFS？

每一轮必须同时删除当前所有叶子，不能沿着某一个方向连续向内删除，否则会破坏不同方向同步向中心收缩的过程。

按层 BFS 的关键是：

```java
int size = queue.size();
```

这个 `size` 是当前层的叶子数量。

随后只处理固定的 `size` 个节点：

```java
for (int i = 0; i < size; i++) {
    int current = queue.poll();

    // 处理中加入 queue 的节点属于下一层
}
```

当前层处理中，新发现的叶子被加入队尾，但不会在本轮处理；下一次进入 `while` 时，它们才成为新的当前层。

这和双队列写法完全等价：

```text
currentQueue → 当前层
nextQueue    → 下一层
```

单队列中的对应关系是：

```text
队列最前面的 size 个元素 → 当前层
处理中加入队尾的元素      → 下一层
```

## 12. 为什么最后一轮就是答案？

Leaf Trimming 的每一轮都在排除当前最外围的候选根。

越早被删除的节点越靠近树的外围；越晚被删除的节点越靠近树的中心。

所以：

```text
第一轮：最外围叶子
第二轮：次外围节点
第三轮：更内部节点
...
最后一轮：树的中心
```

可以维护所有轮次的节点，并记录最后一轮大小：

```java
round = size;
```

遍历结束后，取结果列表最后 `round` 个元素，就是最后一层留下的中心节点。

另一种常见方式是每轮直接用当前层覆盖结果，这样循环结束时结果自然就是最后一轮。

## 13. 完整代码

```java
class Solution {
    public List<Integer> findMinHeightTrees(int n, int[][] edges) {
        List<Integer> res = new ArrayList<>();
        int[] degrees = new int[n];
        List<List<Integer>> rel = new ArrayList<>();
        Queue<Integer> queue = new ArrayDeque<>();
        int round = 0;

        if (n == 1) {
            return new ArrayList<>(List.of(0));
        }

        for (int i = 0; i < n; i++) {
            rel.add(new ArrayList<>());
        }

        for (int[] edge : edges) {
            int a = edge[0];
            int b = edge[1];

            degrees[a]++;
            degrees[b]++;

            rel.get(a).add(b);
            rel.get(b).add(a);
        }

        for (int node = 0; node < n; node++) {
            if (degrees[node] == 1) {
                queue.offer(node);
            }
        }

        while (!queue.isEmpty()) {
            int size = queue.size();
            round = size;

            for (int i = 0; i < size; i++) {
                int current = queue.poll();
                res.add(current);

                for (int target : rel.get(current)) {
                    if (--degrees[target] == 1) {
                        queue.offer(target);
                    }
                }
            }
        }

        return res.subList(res.size() - round, res.size());
    }
}
```

## 14. 复杂度

设：

```text
V = 节点数
E = 边数
```

树中：

```text
E = V - 1
```

建立邻接表需要访问每条边一次；每个节点只会成为一次叶子并进入队列；每条无向边只会被检查有限次数。

因此：

```text
时间复杂度：O(V + E) = O(n)
空间复杂度：O(V + E) = O(n)
```

相比暴力解法的 `O(n²)`，Leaf Trimming 消除了“为每个候选根重新遍历整棵树”的重复工作。

## 15. Leaf Trimming 与拓扑排序的关系

经典 Kahn 拓扑排序处理有向图：

```text
寻找入度为 0 的节点
→ 逻辑删除相关边
→ 产生新的入度为 0 的节点
```

Leaf Trimming 处理无向树：

```text
寻找 degree 为 1 的叶子
→ 逻辑删除相关边
→ 产生新的 degree 为 1 的叶子
```

它们都使用：

```text
degree + queue + 逻辑删除边
```

但严格来说，Leaf Trimming 不是经典拓扑排序，因为经典拓扑排序定义在有向图上。它是一种与 Kahn 算法结构相似的逐层消除方法。

## 16. 这道题能学到什么？

### 先接受暴力解法

没有最佳思路时，先回答：

```text
如果完全不考虑效率，我会怎么做？
```

暴力解法可以帮助明确题目定义和优化目标。最佳算法经常是从暴力算法的重复工作中推导出来的。

### 从确定不可能最优的候选开始排除

优化不一定是直接找到答案，也可以是不断证明哪些候选不可能成为答案。

本题就是：

```text
外围叶子不可能优于内部邻居
→ 先排除所有叶子
→ 在缩小后的问题上重复
```

### 图片布局不代表数据结构

树画在上面还是下面、选择谁作为视觉上的根，都不改变无向边。

判断节点性质时，应依赖：

```text
邻接关系
degree
路径距离
```

而不是依赖图片中的位置。

### `degree + queue` 是一种通用消除模式

当一个节点满足某个 degree 条件时进入队列；处理它以后，更新相关节点的 degree，并产生新的可处理节点。

这个模式会出现在：

- 拓扑排序；
- 课程依赖；
- 最终安全节点；
- Leaf Trimming；
- 图的核心结构剥离。

### 按层 BFS 的本质是保存边界

使用：

```java
int size = queue.size();
```

不是固定语法，而是在明确区分：

```text
当前轮已经存在的节点
当前轮中新产生、下一轮才处理的节点
```

单队列保存 `size` 与双队列保存 `currentQueue/nextQueue`，本质完全相同。

### 不一定要真实修改数据结构

从邻接表中真正删除边比较麻烦。本题用 degree 减一表示逻辑删除，既保留原始关系，又能表达当前剩余结构。

这是图算法中非常值得迁移的技巧。

## 17. 做题推导链条

```text
根的高度 = 根到最远节点的距离
→ 暴力：每个节点作为根，分别遍历整棵树
→ 暴力会重复访问相同节点和边
→ 决定高度的最远节点一定是叶子
→ 外围叶子不可能优于它唯一的邻居
→ 可以同时排除当前所有叶子
→ 删除一层后，内部候选的相对优劣不变
→ 不断从外围向内部收缩
→ 最后一轮剩下树的一个或两个中心
```

## 18. 一句话总结

> Minimum Height Trees 的核心不是反复计算每个根的高度，而是利用 Leaf Trimming 同时排除所有外围叶子，通过按层 BFS 不断向内收缩，最后留下树的中心。
