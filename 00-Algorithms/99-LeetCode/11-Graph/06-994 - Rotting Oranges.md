# LeetCode 994：Rotting Oranges

> 难度：Medium  
> 主题：网格图（Grid Graph）、多源 BFS、按层扩散

## 1. 题目在问什么？

给定一个二维网格：

```text
0：空格子
1：新鲜橘子
2：腐烂橘子
```

每一分钟，所有腐烂橘子都会同时让上下左右相邻的新鲜橘子腐烂。

题目要求的是：

```text
所有新鲜橘子腐烂需要多少分钟？
若有新鲜橘子永远无法腐烂，返回 -1。
```

重点不在于“一个橘子如何扩散”，而在于：**同一分钟内，所有腐烂橘子同时扩散。**

## 2. 这题在训练什么？

这题训练的是 BFS 最重要的一个用途：

> 按距离、按轮次、按时间一层层向外扩散。

下列描述通常都在提示你使用这种 BFS：

```text
最少多少步
最少多少次操作
第几分钟传播完成
每一轮同时扩散
```

在本题中：

```text
BFS 的一层 = 过去一分钟。
```

这与 [[02-102-Binary Tree Level Order Traversal]] 的核心是同一件事：

```text
102：一层 = 二叉树的一层节点。
994：一层 = 一分钟内同时扩散的橘子。
```

## 3. 为什么是多源 BFS？

普通 BFS 通常只有一个起点。

本题可能一开始就有多个腐烂橘子；它们必须同时开始传播，所以每一个初始值为 `2` 的格子都是起点。

```text
扫描 grid
→ 找到所有 2
→ 全部先放入 BFS 队列
→ 从它们一起向外扩散
```

如果只找到一个腐烂橘子再开始遍历，会把“同时发生”的过程误写成“一个接一个发生”。

## 4. 这次思考过程里最关键的转变

一开始容易把问题理解成：

```text
找一个烂橘子
→ 让它扩散
→ 再找下一个烂橘子
```

但正确的时间模型是：

```text
当前所有烂橘子
→ 同时感染一批新鲜橘子
→ 这一批橘子在下一分钟才继续感染别人
```

因此，不能让一个刚腐烂的橘子在同一分钟内继续扩散；否则时间会被压缩，BFS 的“按层”意义就消失了。

## 5. 你的双队列思路在表达什么？

你的写法使用：

```text
queue：本分钟正在扩散的腐烂橘子
temp：本分钟新腐烂、下一分钟才能扩散的橘子
```

逻辑是：

```text
处理 queue 中全部坐标
→ 期间把新腐烂橘子放进 temp
→ queue 清空，说明这一分钟结束
→ 若 temp 不为空，时间加一
→ temp 移入 queue，开始下一分钟
```

这是一个正确的分层 BFS 模型。它把“当前层”和“下一层”分开，因此很直观地保证了同步扩散。

## 6. 计时问题：什么时候才能加一分钟？

这是这题最容易错的地方。

`queue` 被处理完，只代表当前这一层已经结束；并不必然代表真的发生了一分钟的扩散。

例如只剩下无法继续感染别人的腐烂橘子时：

```text
queue 被清空
但没有任何新鲜橘子进入下一层
```

此时不能增加时间。

你后面加入的判断体现了正确原则：

> 只有下一层存在新橘子时，才说明发生了一次有效扩散，分钟数才增加。

## 7. BFS 结束不等于任务完成

这是这次调试中最重要的检查点。

对于下面这种情况：

```text
[[2,1,1],
 [0,1,1],
 [1,0,1]]
```

可以被感染的部分确实会扩散 4 分钟，但左下角的新鲜橘子被 `0` 隔开，永远无法被访问。

所以：

```text
队列为空
≠ 所有新鲜橘子都腐烂了
```

队列为空的真实含义只是：

```text
从所有腐烂源出发，已经没有新的可达新鲜橘子。
```

因此结束后还要扫描一次网格，检查是否还保留 `1`。这一步是在验证题目的最终条件，而不是 BFS 本身的一部分。

## 8. `visited` 的理解

这题里 `visited` 的目的不是 BFS 专属；DFS 在图中也同样需要它。它的作用是：

```text
一个格子一旦已经被发现，就不应再次进入待处理结构。
```

你的代码在橘子加入 `temp` 时立即标记：

```text
发现新鲜橘子
→ 标记 visited
→ 放入 temp
```

这是好习惯。因为同一个新鲜橘子可能同时邻接多个腐烂橘子；如果晚到出队时才标记，它可能会被重复加入。

另外，本题也能直接把格子从 `1` 改为 `2` 来承担“已经发现”的状态，所以 `visited` 和修改 `grid` 的信息有一点重复。保留 `visited` 没有错，只是之后可以练习判断：

```text
题目已有的状态，是否已经足够表示 visited？
```

## 9. 可以进一步改进的地方

你的双队列写法很适合刚学习分层 BFS 时理解“当前分钟”和“下一分钟”。下一步可以练习更常见的单队列写法：

```text
每一轮开始，保存 queue 当前大小 size。
只取出 size 个元素。
这一轮中新加入 queue 的元素，天然属于下一分钟。
```

也就是说：

```text
双队列：用 temp 明确隔开当前层和下一层。
单队列 + size：用固定的 size 隔开当前层和下一层。
```

两者本质相同。先能正确写出双队列，再理解 `size = queue.size()`，这个学习顺序很好。

代码层面还有两个小习惯可以继续保持：

- 行、列建议统一命名为 `nextR`、`nextC`，不要将列写成 `nextY`；
- 发现无法满足最终条件时，直接返回结果会比先给 `res` 赋值再返回更清晰。

## 10. 这题和已经做过的题如何区分？

| 题目 | BFS / DFS 在做什么 | 计数含义 |
| --- | --- | --- |
| 733 Flood Fill | 从一个起点填满连通区域 | 不需要按层计时 |
| 200 Number of Islands | 找到每一块连通陆地 | 每发现一块新岛加一 |
| 102 Binary Tree Level Order Traversal | 按树层收集节点 | 每轮收集一层结果 |
| 994 Rotting Oranges | 多个起点按时间同步扩散 | 每轮代表一分钟 |

选择 BFS 或 DFS 不能只看“这是网格题”，而要看题目问什么：

```text
统计连通区域：DFS / BFS 都可以。
求最少步数、轮数、传播时间：优先考虑 BFS 分层。
```

## 11. 做这类题时的检查清单

```text
节点是什么？
相邻关系是什么？是否只允许上下左右？
起点有一个还是多个？
题目是否强调“同时”“最少步数”“第几分钟”？
一轮 BFS 在题目中代表什么？
新发现的节点是否在入队时就标记？
队列结束后，是否还需要验证最终条件？
```

## 12. 一句话总结

> Rotting Oranges 的重点不是“把橘子变成 2”，而是把“所有腐烂橘子每分钟同时扩散”翻译成多源、分层 BFS；并在扩散结束后确认是否仍有无法到达的新鲜橘子。
