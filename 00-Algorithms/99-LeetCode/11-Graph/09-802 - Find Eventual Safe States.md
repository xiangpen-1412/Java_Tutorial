# LeetCode 802：Find Eventual Safe States

> 难度：Medium  
> 主题：有向图、环、反向图、出度、拓扑排序

## 1. 题目在问什么？

输入 `graph` 是一张有向图的邻接表：

```java
graph[i]
```

表示从节点 `i` 出发，可以直接到达哪些节点。

例如：

```java
graph[2] = new int[]{3, 5};
```

表示：

```text
2 → 3
2 → 5
```

如果：

```java
graph[5].length == 0
```

说明节点 `5` 没有出边，是一个终点节点（terminal node）。

题目要求找出所有安全节点，并按节点编号升序返回。

安全节点的定义是：

> 从该节点出发，无论每一步选择哪条边，最终都一定会到达终点。

## 2. 最关键的逻辑词：所有路径

题目要求的是：

```text
每一条可能的路径最终都能结束
```

而不是：

```text
至少存在一条可以到达终点的路径
```

例如：

```text
      → T（终点）
X →
      → A → B → A
```

虽然从 `X` 可以走到终点 `T`，但也可以进入 `A、B` 构成的环，并永远绕下去。

只要存在一条不能结束的路径，就足以说明 `X` 不是安全节点。

这是处理“所有、任意、一定、无论如何”类条件时的通用思维：

```text
要推翻“所有情况都成立”，只需要找到一个反例。
```

## 3. 安全节点与环的关系

图中的节点数量有限。如果一条路径能够无限走下去，那么它迟早会重复访问某个节点，从而形成环。

因此：

```text
节点不安全
⇔ 从该节点出发，存在一条路径可以进入某个环
```

不安全的节点包括：

1. 环中的节点；
2. 本身不在环里，但能够沿有向边进入环的节点。

例如：

```text
X → Y → A → B
        ↑   ↓
        └───┘
```

`A、B` 位于环中，`X、Y` 能够进入环，所以它们都可能产生永不结束的路径。

自环也属于环：

```text
A → A
```

## 4. “能够到达环”和“被环到达”

有向图只能顺着箭头移动，因此以下两种关系不同。

### 节点能够到达环

```text
X → ... → 环
```

环位于 `X` 的未来。从 `X` 出发存在进入环的可能，所以这个环会影响 `X` 的安全性。

### 节点能被环到达

```text
环 → ... → Y
```

环位于 `Y` 的过去。从 `Y` 出发不能沿箭头倒着回到环，因此这个环本身不能说明 `Y` 不安全。

例如：

```text
X → A ⇄ B → Y
```

- `X` 能够到达环，是环的上游节点；
- `Y` 能被环到达，是环的下游节点；
- 判断 `Y` 时，只需要研究从 `Y` 出发以后能到哪里。

最重要的判断原则是：

> 判断一个节点是否安全，只看它的未来，不看以前有哪些节点能够来到这里。

当然，如果 `Y` 之后还能进入另一个环，那么它会因为第二个环而不安全。

## 5. 为什么从终点反向思考？

终点节点没有出边，因此它一定安全。

如果一个节点的所有出边都通向已经确认安全的节点，那么这个节点也安全。

例如：

```text
A → B
A → C
```

只有当 `B` 和 `C` 都已经确认安全时，才能确认 `A` 安全。

这正好对应题目要求的“所有路径”。只确认其中一个方向还不够。

因此可以从最确定的终点开始，沿着边的反方向逐步判断它们的前驱节点。

## 6. 为什么需要反向图？

原图：

```java
graph[from]
```

只能快速回答：

```text
from 指向谁？
```

但从安全终点向前驱传播时，需要快速回答：

```text
谁指向 current？
```

所以建立反向图：

```text
原边：from → to
反向记录：reverse[to] 中加入 from
```

代码：

```java
List<List<Integer>> reverse = new ArrayList<>();

for (int i = 0; i < graph.length; i++) {
    reverse.add(new ArrayList<>());
}

for (int from = 0; from < graph.length; from++) {
    for (int to : graph[from]) {
        reverse.get(to).add(from);
    }
}
```

例如原图是：

```text
0 → 1
0 → 2
1 → 2
```

反向图保存：

```text
reverse[0] = []
reverse[1] = [0]
reverse[2] = [0, 1]
```

因此处理节点 `2` 时，可以直接找到它的前驱 `0、1`，不必重新扫描整张图。

## 7. 为什么只修改出度，不删除原图的边？

数组中的元素不适合频繁删除，而且真正需要记录的不是“边是否还存在”，而是：

```text
每个节点还有多少条尚未确认安全的出边？
```

使用：

```java
int[] outDegree = new int[graph.length];
```

初始化：

```java
outDegree[i] = graph[i].length;
```

假设：

```text
A → B
A → C
```

最开始：

```text
outDegree[A] = 2
```

确认 `B` 安全后，在逻辑上消除 `A → B`：

```text
outDegree[A]: 2 → 1
```

确认 `C` 安全后：

```text
outDegree[A]: 1 → 0
```

当剩余出度变成 `0` 时，说明 `A` 的所有出边都已通向确认安全的节点，因此 `A` 也可以加入待处理队列。

运行过程中：

- 原始 `graph` 不修改；
- `reverse` 不修改；
- 只动态减少 `outDegree`；
- 队列保存当前剩余出度为 `0`、等待向前驱传播的节点。

## 8. 完整算法：反向图 + Kahn 拓扑消除

```java
class Solution {
    public List<Integer> eventualSafeNodes(int[][] graph) {
        int n = graph.length;

        int[] outDegree = new int[n];
        List<List<Integer>> predecessors = new ArrayList<>(n);

        for (int i = 0; i < n; i++) {
            predecessors.add(new ArrayList<>());
        }

        // 保存原图出度，并建立反向图
        for (int node = 0; node < n; node++) {
            outDegree[node] = graph[node].length;

            for (int next : graph[node]) {
                predecessors.get(next).add(node);
            }
        }

        Queue<Integer> queue = new ArrayDeque<>();

        // 所有终点节点最先确认安全
        for (int node = 0; node < n; node++) {
            if (outDegree[node] == 0) {
                queue.offer(node);
            }
        }

        // 从安全节点出发，反向消除其前驱的一条出边
        while (!queue.isEmpty()) {
            int safeNode = queue.poll();

            for (int previous : predecessors.get(safeNode)) {
                outDegree[previous]--;

                if (outDegree[previous] == 0) {
                    queue.offer(previous);
                }
            }
        }

        // 按节点编号扫描，结果天然升序
        List<Integer> answer = new ArrayList<>();

        for (int node = 0; node < n; node++) {
            if (outDegree[node] == 0) {
                answer.add(node);
            }
        }

        return answer;
    }
}
```

## 9. 为什么最后 `outDegree == 0` 就是安全节点？

算法结束后：

- 终点节点的出度原本就是 `0`；
- 所有路径都通向安全区域的节点，会被逐层消除到 `0`；
- 环中节点至少会保留一条指向环内的边；
- 能够进入环的节点也会保留至少一条无法消除的出边。

因此：

```text
剩余出度为 0 → 安全节点
剩余出度大于 0 → 能够到达某个环，不安全
```

最后从 `0` 扫描到 `n - 1`，可以直接得到升序答案，不需要额外排序。

## 10. 为什么不用 `int[][]` 保存反向图？

反向图中每个节点的前驱数量一开始未知，而普通数组需要提前确定长度。

最方便的选择是：

```java
List<List<Integer>> reverse
```

因为内部 `ArrayList` 可以动态增长：

```java
reverse.get(to).add(from);
```

如果必须使用 `int[][]`，则需要：

```text
第一遍统计每个节点的入度
→ 按入度创建每个内部数组
→ 第二遍填写前驱节点
```

这会增加实现复杂度，所以本题使用嵌套 `ArrayList` 更自然。

## 11. 复杂度

设：

```text
V = 节点数
E = 边数
```

建立反向图会访问每条边一次，拓扑消除也只会处理每条边一次：

```text
时间复杂度：O(V + E)
空间复杂度：O(V + E)
```

由于任何正确算法至少都需要读取全部节点和边，`O(V + E)` 已经是渐进意义上的最优时间复杂度。

## 12. 常见错误

### 错误一：存在一条路径到终点就认为安全

题目要求所有路径都能结束。只要有一个分支可以进入环，节点就不安全。

### 错误二：只标记环中的节点

环的上游节点虽然不在环内，但能够进入环，同样不安全。

### 错误三：把“能够到达环”和“被环到达”混淆

判断节点时只看从该节点出发的未来。环能走到某个下游节点，不代表该节点还能返回环。

### 错误四：每次弹出节点后重新扫描整个图

这样会产生大量重复查找。反向图可以直接找到当前节点的所有前驱。

### 错误五：重复把原始终点加入队列

终点只在初始化时入队。之后只有某个前驱的剩余出度第一次变成 `0` 时，才将它加入队列。

### 错误六：真正删除 `int[][]` 中的边

不需要修改原图。用剩余出度减一表示这条边已经完成处理即可。

### 错误七：忘记结果必须升序

队列的处理顺序不保证升序。最简单的方法是在算法结束后，按节点编号扫描 `outDegree`，天然得到有序结果。

## 13. 做题思维总结

遇到类似问题，可以按以下顺序思考：

```text
“所有路径都结束”
→ 反面是“存在一条路径永不结束”
→ 有限图中永不结束意味着重复访问
→ 重复访问意味着进入环
→ 环及所有能进入环的上游节点都不安全
→ 从确定安全的终点开始反向传播
→ 用反向图快速寻找前驱
→ 用剩余出度代替实际删除边
```

## 14. 一句话总结

> 从所有出度为 `0` 的终点出发，在反向图中逐层消除安全边；最终剩余出度为 `0` 的节点，就是所有最终安全节点。
