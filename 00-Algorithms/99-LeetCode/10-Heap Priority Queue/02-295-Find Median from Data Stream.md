# 02. Find Median from Data Stream：从题目学到什么

## 这题真正训练的能力

这题表面上是在求中位数，真正训练的是：

```text
不要每次都重新处理全部数据；
而是维护一个“随时可回答”的数据结构。
```

数据不断进来时，不能每加入一个数就重新排序一次。要思考：

```text
为了得到中位数，我真正需要知道哪些信息？
```

答案不是完整的排序结果，而是排序后中间位置附近的一个或两个数。

## 从“中位数”倒推需要维护什么

将排好序的数据从中间切开：

```text
较小的一半 | 较大的一半
```

中位数只与分界线两侧最靠近的元素有关：

```text
较小一半的最大值 | 较大一半的最小值
```

所以问题被转化为：

```text
如何持续维护两组数，
并且能快速拿到左边的最大值和右边的最小值？
```

这就是双堆出现的原因：

```text
small：保存较小的一半，用大顶堆，堆顶是左边最大值
large：保存较大的一半，用小顶堆，堆顶是右边最小值
```

这是一种很重要的解题方式：

```text
先找答案依赖的“边界信息”，
再选择恰好能维护这些边界信息的数据结构。
```

## 双堆不是重点，不变量才是重点

堆只是工具，真正必须一直成立的是两个不变量。

### 1. 左边不能出现比右边更大的数

```text
small 中任意元素 <= large 中任意元素
```

否则两边就不再对应“较小的一半”和“较大的一半”。

### 2. 两边数量必须接近

```text
small.size() == large.size()
或
small.size() == large.size() + 1
```

这里约定让 `small` 在奇数个元素时多一个。

只要这两件事始终正确，中位数就不需要再计算或排序：

```text
一样多：取两个堆顶的平均值
small 多一个：取 small 的堆顶
```

## 插入一个新数时的思考顺序

每次 `addNum(num)`，不要只想着“把数放进堆里”。应该按不变量思考：

```text
1. 它属于左半边还是右半边？
2. 放进去后，左右数量是否还符合规则？
3. 如果不符合，移动哪一个边界元素才能恢复规则？
```

具体表现为：

```text
num <= small.peek()：放入 small
否则：放入 large

small 太多：small.poll() 移到 large
large 太多：large.poll() 移到 small
```

这里移动的是堆顶，而不是任意元素。因为堆顶刚好是跨越分界线时最合适的元素：

```text
small.poll()：左边最大的数，移到右边后仍保持顺序
large.poll()：右边最小的数，移到左边后仍保持顺序
```

## 从原代码中学到的调试方法

### 先检查初始状态

```java
if (num < small.peek())
```

第一次加入数字时，`small` 为空，`peek()` 得到 `null`。因此设计数据结构题时，先问自己：

```text
空集合时每一个操作是否安全？
```

常见保护写法：

```java
if (small.isEmpty() || num <= small.peek())
```

### “平衡”不等于“完全相等”

中位数允许奇数个元素，因此两个堆的大小差应当最多为 `1`，不能每次都强行调到相同大小。

这个错误提醒我们：

```text
写 balance() 前，先把“允许的状态”精确写出来。
```

本题允许的是：

```text
(small, large) = (k, k) 或 (k + 1, k)
```

而不是只有：

```text
(k, k)
```

### 返回类型是 `double`，表达式也要是浮点计算

```java
return (small.peek() + large.peek()) / 2;
```

即使方法返回 `double`，上面的 `/ 2` 也会先做整数除法。例如 `3 / 2` 先得到 `1`，最后才变成 `1.0`。

应写：

```java
return ((long) small.peek() + large.peek()) / 2.0;
```

这说明：

```text
函数返回类型不会改变表达式内部的计算规则。
要在计算发生之前，就让其中一个操作数成为 double。
```

### 自动转换的实际含义

奇数个元素时可以写：

```java
return small.peek();
```

因为 Java 会自动：

```text
Integer -> int -> double
拆箱       类型提升
```

可以把它记成：

```text
小转大通常自动；大转小需要手动。
```

但自动拆箱有前提：包装类不能是 `null`，否则会发生 `NullPointerException`。

## 以后遇到类似题目的识别信号

看到下面这些描述，可以考虑双堆：

```text
数据持续加入 / 在线处理（stream）
需要动态查询中位数、百分位数或第 k 个附近的值
答案由“中间分界线”两侧的元素决定
```

但要注意：双堆特别适合“只插入 + 查询”。如果题目还要求任意删除历史元素，就需要额外的延迟删除、哈希表计数，或其他平衡树结构。

## 可复用的思考框架

```text
1. 每次查询真正依赖哪些信息？
2. 能否把全部数据切成两部分？
3. 分界线处需要快速得到哪个最大值或最小值？
4. 用什么结构维护这些边界？
5. 插入或删除后，哪些不变量必须恢复？
```

这题的答案是：

```text
中位数依赖中间边界；
把数据分成左右两半；
维护左最大、右最小；
用大顶堆 + 小顶堆；
始终保证顺序正确、数量差最多为 1。
```

## 最后记住

```text
数据流题的关键不是“把数据存下来”，
而是“持续维护下一次查询所需的最少信息”。
```
