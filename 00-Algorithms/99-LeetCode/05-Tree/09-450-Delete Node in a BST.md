# 450. Delete Node in a BST

> 难度：Medium  
> 主题：Binary Search Tree、递归契约、子树重建、中序后继

## 问题

给定一棵二叉搜索树的根节点 `root` 和整数 `key`，删除值为 `key` 的节点，并返回删除后的树根。若树中不存在该值，则保持原树不变。

难点不在利用 BST 找到目标，而在删除后仍要保持 BST 的结构：叶子节点会消失，单孩子节点要由孩子接替，双孩子节点则要选择一个合法值接替当前位置。

## 核心收获

- **识别信号**：题目不仅查询树，还可能改变父子连接，因此递归不能只负责“找到并删除”，还要把修改后的子树入口交还给父层。
- **核心契约**：`deleteNode(root, key)` 表示“删除以 `root` 为根的子树中的 `key`，并返回处理后的新子树根”。返回值可能是原 `root`、某个孩子或 `null`。
- **问题缩小与结果重建**：向下递归时，大树被拆成越来越小的子树；向上返回时，父层接住新子树根并重新连接，最终恢复成完整树。
- **双孩子转换**：用右子树最小值（中序后继）覆盖当前值，再从右子树删除该后继。后继没有左孩子，因此第二次删除一定退化为零孩子或单孩子问题。

## `return root` 的设计含义

树递归不是只处理一个孤立节点，而是把每个节点视为一棵子树的入口：

```text
输入：以 root 为入口的旧子树
输出：处理完成后的新子树入口
```

当目标位于左边时：

```java
root.left = deleteNode(root.left, key);
```

可拆成：

```text
把左子树交给递归处理
→ 得到处理后的左子树新根
→ 将新根重新接到 root.left
```

左子树可能换根甚至变为 `null`，但当前节点没有被删除，所以当前整棵子树的根仍然是 `root`：

```java
return root;
```

它不是固定收尾，而是在回答：

> 当前子树处理完以后，对外暴露的根仍然是谁？

只有当前根被删除且需要由孩子接替时，返回值才会改变：

```java
if (root.left == null) return root.right;
if (root.right == null) return root.left;
```

## 删除当前根的三种情况

### 零个或一个孩子

```text
左孩子为空 → 返回右孩子
右孩子为空 → 返回左孩子
```

若当前节点是叶子，左右孩子都为空，第一条会直接返回 `null`，因此叶子不需要单独分支。

例如：

```text
父节点.left → 3 → 5
```

删除 `3` 后，递归返回 `5`，上一层执行：

```text
父节点.left = 5
```

当前递归层不必知道父节点是谁，也不必记录自己是左孩子还是右孩子；返回新根就能把结构变化通知父层。

### 两个孩子

例如删除 `5`：

```text
        5
       / \
      3   9
         / \
        7  11
```

右子树最小值为 `7`。先把当前值改为 `7`，再删除右子树中原来的 `7`：

```text
        7
       / \
      3   9
           \
           11
```

选择右子树最小值是因为它：

- 大于左子树中的所有值；
- 小于等于右子树中的其他值；
- 没有左孩子，删除原位置时结构简单。

也可以对称地使用左子树最大值（中序前驱）。

## 结构骨架

```java
if (root == null) return null;

if (key < root.val) {
    root.left = deleteNode(root.left, key);
} else if (key > root.val) {
    root.right = deleteNode(root.right, key);
} else {
    if (root.left == null) return root.right;
    if (root.right == null) return root.left;

    TreeNode successor = findMin(root.right);
    root.val = successor.val;
    root.right = deleteNode(root.right, successor.val);
}

return root;
```

复习时不要背分支顺序，先问每个分支同一个问题：

> 删除完成后，当前这棵子树的新根是谁？

## 递归过程：向下缩小，向上重建

```text
deleteNode(整棵树)
    ↓ 根据 BST 大小关系
deleteNode(目标所在子树)
    ↓
deleteNode(更小的子树)
    ↓ 找到目标并得到局部新根
    ↑ 父层重新连接 left/right
    ↑ 返回当前子树根
    ↑
返回整棵树的新根
```

因此最外层也必须接住返回值：

```java
root = deleteNode(root, key);
```

否则删除整棵树的根时，调用者不会得到新的根节点。

## 变种与注意点

- `findMin(root.right)` 只能在确认右子树非空后调用；若遗漏零孩子和单孩子分支，递归删除叶子或后继时会把 `null` 传给 `findMin`，触发 `NullPointerException`。
- 该实现复制的是后继的 `val`，没有移动节点对象。若节点还承载不可分割的业务数据，必须同步替换完整数据或真正调整节点引用。
- 代码通常默认键值唯一；若允许重复值，必须先定义重复值放置和删除的规则。
- 时间复杂度为 `O(h)`，递归栈为 `O(h)`；平衡 BST 中 `h = O(log n)`，退化成链表时 `h = O(n)`。

## 我的思维误区

- **把递归理解成“进入树里执行删除动作”**：这种视角难以处理父节点如何重新接线，还容易想引入 `parent` 和“左/右孩子”标记。替代判断是先写契约：函数接收一棵子树，返回修改后的子树根。
- **认为 `return root` 是突然出现的技巧**：实际上它是契约的直接结论。若只修改了当前根下面的结构，当前子树入口仍是 `root`；若当前根被孩子替代，才返回对应孩子或 `null`。
- **默认找到目标后一定能找右子树最小值**：只有两个孩子都存在时才需要后继。零孩子和单孩子必须先返回替代根，否则会对 `null` 调用 `findMin`。

## 一句话总结

> 把删除函数定义为“返回处理后的子树新根”：向下利用 BST 定位目标，局部决定新根，向上逐层重新连接；`return root` 表示当前节点仍是这棵已处理子树的入口。
