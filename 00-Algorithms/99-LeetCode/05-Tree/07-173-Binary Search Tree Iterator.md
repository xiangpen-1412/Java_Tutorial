# 173. Binary Search Tree Iterator

> 难度：Medium  
> 主题：Binary Search Tree、中序遍历、迭代器、栈

## 问题

给定一棵二叉搜索树，实现一个按中序遍历顺序工作的迭代器：

- `next()`：把指针移到下一个节点并返回节点值；
- `hasNext()`：只判断是否还有下一个节点，不能消耗节点；
- 初始指针位于所有元素之前，因此第一次 `next()` 返回 BST 的最小值。

题目保证调用 `next()` 时一定存在下一个节点。关键不只是得到一次中序遍历结果，而是保存遍历进度，让多次方法调用共同完成遍历。

## 核心收获

- **识别信号**：BST 满足“左小右大”，中序遍历按“左、根、右”访问，因此访问结果天然升序。题目又要求逐个取下一个值，所以应把中序遍历理解成一个可以暂停和恢复的过程。
- **状态与查询分离**：`next()` 负责推进状态；`hasNext()` 只观察状态。查询方法如果偷偷修改索引或栈，会造成节点被跳过。
- **简单做法**：先完整中序遍历，把节点保存到列表，再用索引表示当前位置。构造耗时和额外空间都是 `O(n)`，之后 `next()`、`hasNext()` 都是 `O(1)`。
- **进阶做法**：用栈保存“已经走到、但尚未访问”的祖先，只展开当前需要的左链。额外空间为 `O(h)`，其中 `h` 是树高；单次 `next()` 最坏为 `O(h)`，但每个节点只进栈、出栈一次，因此连续调用的均摊时间为 `O(1)`。

## 列表方案中的索引含义

假设中序结果是：

```text
3, 7, 9, 15, 20
```

初始化：

```text
index = -1
```

`index` 表示“最近一次已经返回的位置”：

```text
next()    → index 增加后读取当前位置
hasNext() → 检查 index + 1 是否仍在列表范围内
```

合法下标为 `0` 到 `size - 1`，所以判断的本质是：

```text
下一个下标 < 元素数量
```

## 栈方案为什么成立

中序遍历必须先访问一棵子树最左侧的节点。因此初始化时，可以把从根到最左节点的整条路径压栈：

```text
        7
       / \
      3   15
         /  \
        9   20

初始栈：7, 3（3 在栈顶）
```

栈顶就是当前尚未访问的最小节点。弹出一个节点后：

1. 该节点本身已经访问；
2. 它的左子树已经处理完；
3. 接下来轮到它的右子树；
4. 为了先访问右子树中的最小值，需要把右孩子开始的整条左链压栈。

可记住下面的结构，而不必记完整代码：

```text
初始化：压入 root 的整条左链
next：弹出栈顶；压入其右子树的整条左链
hasNext：栈是否非空
```

## 变种与注意点

- 如果只调用少量几次 `next()`，完整列表方案仍会预先遍历整棵树；栈方案只展开目前需要的路径。
- 平衡树中 `h = O(log n)`；BST 退化为链表时 `h = O(n)`。
- `hasNext()` 应当是无副作用的查询。连续调用多次 `hasNext()`，后续 `next()` 返回的值不应改变。
- 保存 `TreeNode` 或只保存 `int` 都能完成列表方案；若只需要返回值，保存整数表达更直接。

## 我的思维误区

- 曾在 `hasNext()` 中使用 `++index`。这会在判断之前推进索引，使一次正常的调用序列 `next() → hasNext() → next()` 跳过元素，最后还可能访问 `res.get(size)` 而越界。
- 替代判断：先明确状态变量含义。若 `index` 是“最后已返回元素的位置”，那么 `hasNext()` 只应检查 `index + 1` 是否存在，只有 `next()` 才能改变 `index`。

## 一句话总结

> BST 的中序遍历提供升序顺序；迭代器的关键是保存“遍历到哪里”，并严格区分推进状态的 `next()` 与只读查询的 `hasNext()`。
