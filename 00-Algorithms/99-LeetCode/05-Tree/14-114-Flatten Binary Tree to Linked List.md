# 114. Flatten Binary Tree to Linked List

> 难度：Medium  
> 主题：Binary Tree、前序遍历、原地修改、反向递归、指针重连

## 问题

给定一棵二叉树，原地将它展开为一条只使用 `right` 指针的链表。展开后的节点顺序必须与原树的**前序遍历**一致，并且每个节点的 `left` 都必须为 `null`。

```text
原树：

        1
       / \
      2   5
     / \   \
    3   4   6

展开：

1 → 2 → 3 → 4 → 5 → 6
```

题目中的“链表”仍然由原来的 `TreeNode` 节点组成，不需要创建新的链表节点。

## 核心收获

- **识别信号**：题目明确要求展开顺序与前序遍历一致，因此目标顺序就是“根、左、右”。
- **核心转换**：可以先得到前序节点序列再重连；进一步优化时，可以反向处理前序序列，让每个节点直接连接已经构造好的后半段。
- **通用方法**：修改树指针前必须考虑原来的孩子是否还需要访问。先收集后修改不会破坏遍历；原地递归则通过“右、左、根”的反向顺序避免丢失节点。

## 我的方案：前序收集后统一重连

先用前序遍历把节点引用保存到列表：

```text
[1, 2, 3, 4, 5, 6]
```

然后把每对相邻节点连接起来：

```java
node.left = null;
node.right = next;
```

### 为什么这个方案正确

`List<TreeNode>` 保存的是原节点的**引用**，不是节点副本。因此从列表中取出节点并修改：

```java
TreeNode node = nodes.get(i);
node.right = nodes.get(i + 1);
```

实际修改的就是原二叉树中的节点。

而且指针重连发生在完整的前序遍历之后。此时所有节点已经保存到列表，即使后续覆盖原来的 `left` 和 `right`，也不会再影响节点收集。

原实现只循环到倒数第二个节点：

```java
for (int i = 0; i < nodes.size() - 1; i++)
```

这仍然可以通过，因为前序遍历的最后一个节点一定是叶子：如果它还有孩子，孩子就会在它之后被访问，它便不可能是最后一个节点。不过，更稳妥且语义更完整的写法是显式处理最后一个节点：

```java
for (int i = 0; i < nodes.size(); i++) {
    TreeNode node = nodes.get(i);
    node.left = null;
    node.right = i + 1 < nodes.size()
            ? nodes.get(i + 1)
            : null;
}
```

这一方案的复杂度为：

- 时间：`O(n)`
- 节点列表：`O(n)`
- 递归栈：`O(h)`

它直观地体现了“目标链表就是前序遍历结果”，但没有充分利用可以直接修改树指针的条件。

## 标准递归：反向前序遍历

正常前序顺序是：

```text
根 → 左 → 右
```

如果从最终链表的尾部向前构造，处理顺序应该反过来：

```text
右 → 左 → 根
```

维护一个 `previous`，表示已经构造好的链表头。

### 递归不变量

处理当前 `root` 时：

> `previous` 已经指向前序遍历中位于 `root` 后面的所有节点组成的链表。

因此当前节点只需要执行：

```java
root.right = previous;
root.left = null;
previous = root;
```

结构骨架：

```java
TreeNode previous;

void flatten(TreeNode root) {
    if (root == null) return;

    flatten(root.right);
    flatten(root.left);

    root.right = previous;
    root.left = null;
    previous = root;
}
```

必须先递归右子树，再递归左子树。因为这里不是按正常前序顺序访问，而是在倒着构造最终的前序链表。

## 为什么反向处理更安全

如果刚到当前节点就直接执行：

```java
root.right = root.left;
root.left = null;
```

原来的右子树入口可能会被覆盖，后续无法继续访问。

反向递归会先处理完原来的右子树和左子树，之后才修改当前节点的指针。需要保存的信息已经通过 `previous` 连接好，因此不会丢失任何子树。

## 复杂度

反向递归版本：

- 时间复杂度：`O(n)`，每个节点只处理一次。
- 显式辅助容器：`O(1)`。
- 递归栈：`O(h)`；平衡树为 `O(log n)`，退化链状树为 `O(n)`。

如果题目严格要求包括调用栈在内的 `O(1)` 额外空间，可以使用迭代指针重连：

```text
当前节点存在左子树时：
1. 找到左子树最右侧节点 predecessor
2. predecessor.right 接上当前节点原来的右子树
3. 当前节点的 right 改为 left
4. 当前节点的 left 置空
5. 沿新的 right 继续处理
```

这种方法不需要列表或递归栈，但指针操作比反向递归更容易写错。

## 变种与注意点

- `preOrder()` 只负责收集节点时不需要返回 `TreeNode`，返回类型可以直接写成 `void`。
- “原地修改”表示复用原节点，不代表绝对不能使用辅助空间；列表方案仍然能够通过，只是空间复杂度不是最优。
- 只要需要覆盖树的孩子指针，就要先确认原子树入口是否已经保存或处理完毕。
- `previous` 表达的不是“遍历中的上一个节点”，而是反向构造过程中已经完成的链表头。

## 我的思维误区

- **不确定列表中的节点是否仍然属于原树**：Java 集合保存的是 `TreeNode` 引用，修改列表元素指向对象的字段，就是修改原节点；只有显式 `new TreeNode(...)` 才会创建副本。
- **认为修改指针一定会破坏遍历**：关键取决于修改发生的时机。先完成全部节点收集再修改是安全的；边遍历边修改时才需要提前保存孩子或采用不会丢失入口的遍历方向。

## 一句话总结

> 展开顺序就是前序遍历；列表方案先记录顺序再重连，反向递归则按“右、左、根”处理，并让每个当前节点接到已经构造好的 `previous` 链表头。
