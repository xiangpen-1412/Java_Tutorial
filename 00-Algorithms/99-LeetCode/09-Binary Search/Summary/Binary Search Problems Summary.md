# Binary Search 已做题目总结：寻找单调边界并安全排除区间

二分查找的核心不是数组必须完全升序，而是搜索区间中存在可以判断、可以排除的单调结构。每轮都要明确：`mid` 属于答案的哪一侧，哪部分可以确定不含答案。

## 2. 二分前先写三句话

```text
搜索区间表示什么？
答案满足什么判断条件？
看到 mid 的结果后，哪一侧一定可以排除？
```

不要只背：

```java
if (...) left = mid + 1;
else right = mid - 1;
```

边界是否保留 `mid`，取决于 `mid` 仍不仍可能是答案。

## 3. 精确查找与边界查找

### 精确查找

704 找具体 target：

```text
相等 → 返回
mid 太小 → 排除左半边和 mid
mid 太大 → 排除右半边和 mid
```

### 左边界/插入位置

35 找第一个大于等于 target 的位置。即使 `nums[mid] >= target`，`mid` 仍可能是答案，因此要保留它并继续向左收缩。

这两类问题的区别是：

```text
精确查找：相等立即结束
边界查找：相等也可能继续寻找更靠左/右的位置
```

## 4. 二维矩阵映射为一维

74 的每一行有序，且下一行开头大于上一行结尾，因此整个矩阵按行展开后是有序数组。

一维下标 `mid` 映射为：

```text
row = mid / columnCount
col = mid % columnCount
```

不用真正复制矩阵，也能进行标准二分。

## 5. 旋转数组：整体无序，但至少一半有序

### Search in Rotated Sorted Array

每轮先判断：

```text
左半边有序，还是右半边有序？
```

再判断 target 是否落在有序半边的值域内，从而排除另一部分。

### Find Minimum

将 `nums[mid]` 与 `nums[right]` 比较：

```text
nums[mid] > nums[right] → 最小值一定在 mid 右侧，排除 mid
nums[mid] <= nums[right] → mid 仍可能是最小值，保留 mid
```

这里展示了“何时排除 mid，何时保留 mid”的标准推理。

## 6. Peak Element：对趋势二分

不需要数组整体有序，只比较相邻趋势：

```text
nums[mid] < nums[mid + 1] → 右侧一定存在峰值
否则 → 左侧包含 mid 一定存在峰值
```

二分需要的是可排除性，不一定是全局升序。

## 7. 答案空间二分

875 不是在数组下标中找某个值，而是在“吃香蕉速度”这个答案范围中搜索。

对速度 `speed` 定义可行性：

```text
能否在 h 小时内吃完？
```

速度越大越容易完成，因此可行性具有单调性：

```text
不可行、不可行、……、可行、可行、……
```

目标是找到第一个可行速度，即最小可行解。

## 8. mid 与边界安全

避免潜在加法溢出：

```java
int mid = left + (right - left) / 2;
```

使用 `mid + 1` 比较趋势时，循环条件必须保证它合法，例如让 `left < right`。

## 9. 选择方法

| 题目信号 | 优先考虑 |
| --- | --- |
| 有序数组找确定值 | 标准二分 |
| 找第一个/最后一个满足条件的位置 | 边界二分 |
| 二维矩阵整体按行有序 | 虚拟一维二分 |
| 旋转有序数组 | 判断有序半边或与 right 比较 |
| 相邻趋势能保证某侧存在答案 | 趋势二分 |
| 求最小可行/最大可行数值 | 答案空间二分 |

## 10. 常见错误

```text
没有定义搜索区间是闭区间还是半开区间；
mid 仍可能是答案却使用 mid ± 1 排除；
循环条件和边界更新规则不配套，造成死循环；
旋转数组直接按整体升序判断；
二维下标映射的行列公式写反；
答案空间中可行性计算使用整数除法向下取整；
速度区间的上界和返回边界含义不清。
```

## 11. 记忆主线

```text
二分不是背模板，而是不断证明一半不可能有答案。
mid 不可能是答案 → 排除 mid。
mid 仍可能是边界答案 → 保留 mid。
数组不必整体有序，只要判断结果具有单调性。
最小可行值 → 在答案空间找第一个 true。
```
